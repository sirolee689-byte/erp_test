# ask-then-execute

本 Skill 管**节奏与定稿**（何时对齐、定稿写什么、口令怎么算数）。**怎么写代码**（最小 diff、不顺手重构）单源见 [`docs/agents/coding-discipline.md`](../../docs/agents/coding-discipline.md)。

用户看到方案、拍板，再执行；Agent 不能自行猜测并直接改代码。

---

## A. 核心规则

### 何时启用本 Skill

| 情况 | 行为 |
|------|------|
| 用户 **Attach 本 Skill** 或说 **`$ask-then-execute`** | 默认**全程**两阶段；按下方档位选定稿模板 |
| **未 Attach**，任务为**轻档** | 不必完整定稿；口头一句「打算改 X」+ `改吧`/`OK` 即可执行（见 coding-discipline「小修小补」） |
| **未 Attach**，任务为**中/重档** | 自动走对应档位定稿，重档须用明确执行口令 |

**Cursor Ask 模式**：系统禁止改文件 = 强制停留在 Ask 阶段，只输出定稿或讨论。

### 任务分级（先判档，再选模板）

| 档位 | 典型任务 | Ask 要多少 | 方案数 | 定稿模板 | 执行口令 |
|------|----------|------------|--------|----------|----------|
| **轻** | 错别字、单行 bug、1–2 处样式/文案、`disabled` 一类小逻辑 | 一句说明「打算改什么」 | 通常 0–1 | 不必套模板 | `改吧` / `OK` / `可以改了` |
| **中** | 单页交互、单接口、不动库、单模块 README | 轻量定稿（§B） | 1 个即可；有取舍时再给 2 个 | 轻量 | `按上面改` / `开始改` / `照这个改` |
| **重** | 动库/DDL、批量、权限、跨模块、BOM/成本、新门禁接口 | 完整定稿（§B）+ D3 检查项 | **至少 2 个**，须推荐其一 | 完整 | **`按定稿实现`** / **`按定稿实现 + 方案A`**（见下方授权规则） |

**判档原则**：涉及删除/覆盖/批量/pass/成本/打印/DDL 任一 → 至少**中档**；涉及库表结构变更或跨模块 → **重档**。不确定时按高一档处理。

### 两阶段定义

**Ask 阶段**：只读代码、查文档、跑只读命令、澄清需求。不写 patch，不改文件，不启停服务，不跑会写库的命令。

- 探索代码：有 CodeGraph 时**先** MCP `codegraph_explore`（v1.1+ 单工具），再按需 Read/Grep；勿调用已下架的 `codegraph_search` 等多工具名。

**Execute 阶段**：用户按**所属档位**明确授权后才开始改代码。未授权时，即使已知道怎么改，也只能输出定稿（待确认）。

触发 Ask 阶段的语义：「如有疑问先问」「禁止自行猜测」「只是讨论」「先给方案」。

### 执行授权清单（按档位）

**轻档 — 以下算作授权：**
- `改吧` / `OK` / `可以改了` / `同意` / `执行` / `开始改` / `照这个改`

**中档 — 以下算作授权：**
- `按上面改` / `开始改` / `照这个改` / `执行` / `可以改了`
- `OK` / `同意`：若上一轮已是**无待拍板项**的轻量定稿，可执行；若有 `[ ]` 待拍板或方案未选，**不算**授权

**重档 — 以下才算授权（须含「定稿」或明确方案）：**
- `按定稿实现` / `按定稿实现 + 方案A` / `按定稿实现 + 方案B`
- `按上面定稿实现` / `就按定稿来`

**重档 — 以下不算授权（须先确认）：**
- 单独 `OK` / `同意` / `可以改了` → 先问一句：「是按定稿开始改代码吗？」
- `定稿` / `先说说` / `逻辑是什么` / `有没有疑问` / `别动代码` / 只是在讨论方案
- 定稿里还有未解决的 `[ ]` 待拍板
- 授权句和定稿冲突时，先问一句确认，不要猜

### 什么时候必须追问

不要猜，先问（最多 1–3 个问题，问完就停）：
- 需求会影响数据删除、覆盖、批量更新、审核状态、成本运算、打印输出等关键结果
- 用户给的例子和真实数据不一致
- 方案有明显业务取舍，但用户没有表达偏好
- 「当前页/全部/勾选项」「覆盖/跳过」「新窗口/当前页」等边界不清楚

### 沟通口吻

单源见 [`.cursorrules`](../../.cursorrules) §1（全中文、大白话、界面主语）；Codex 入口 [`AGENTS.md`](../../AGENTS.md) 仅作索引，不重复展开。

### 反模式（禁止）

- 用户只是让讨论，就直接改代码
- **重档**只给一个方案，却不说明为什么没有其它方案（轻/中档豁免）
- 推荐方案但不说明代价
- 定稿没有写「不做什么」
- 定稿还有待拍板事项就执行
- Execute 阶段超出定稿范围（见 §C「增量定稿」）
- 改了后端却不提醒用户按 `.cursorrules` §14 手动重启 API
- 定稿未列文档目标却擅自大改 CONTEXT.md
- 该写 database_map 却只改了 README
- 文档同步未完成就勾选验收标准
- 把接口实现、SQL 全文塞进 CONTEXT.md
- 在多个文件重复粘贴同一段表结构（database_map 为单源）
- 用 .scratch 当永久规则库，从不晋升到 L2/L3

## B. Ask 阶段：定稿模板

### 轻量定稿（中档默认）

```
## 定稿（待你确认）· 轻量

### 先说结论
（推荐做法一句话）

### 目标
（用户最终会看到什么）

### 范围
- 做：...
- 不做：...

### 待你拍板（如有）
- [ ] ...

### 验收标准
- [ ] ...（含：是否改后端/API 重启、跑哪条 `node --test`、是否浏览器点主路径）
- [ ] 文档：L1 `.../README.md` / 本次不同步 L2/L3

### 执行口令
确认后请回复：**按上面改**（或 **开始改**）。
```

### 完整定稿（重档默认）

**重档**须至少 2 个方案，明确推荐一个并说明理由和代价。只有需求非常明确、没有合理分支时，才可只给一个方案（须说明原因）。

```
## 定稿（待你确认）

### 先说结论
我推荐：方案 X。
原因：...

### 方案选择
#### 方案 A：...
- 效果：...
- 优点：...
- 缺点/代价：...
- 适合：...

#### 方案 B：...
- 效果：...
- 优点：...
- 缺点/代价：...
- 适合：...

### 目标
（一句话说明用户最终会看到什么变化）

### 范围
- 做：...
- 不做：...

### 规则与边界
1. ...

### 待你拍板（如有）
- [ ] 选择方案 A / B / C
- [ ] 其它需要用户确认的问题

### 验收标准
- [ ] ...
- [ ] 有界面改动时：浏览器走主路径（IronBee DevTools，见 `.cursor/rules/ironbee-devtools-use.mdc`）
- [ ] 改表/写 database_map 前：`user-sqlserver` `get_table_schema` 核对字段（`.cursorrules` §13）

### 文档同步
- [ ] **L1 模块 README**（.cursorrules §5 强制）：路径 `...`；更新要点：...
- [ ] **L2 领域**：`CONTEXT.md` 章节 / `docs/bom/...`；写什么：...
- [ ] **L3 数据/架构**：`docs/sql/database_map.md` / `docs/adr/`；写什么：...
- [ ] **本次不同步 L2/L3**，原因：...

### L0 落盘（重档建议）
- [ ] Execute 前将本定稿摘要写入 `.scratch/<feature>/定稿.md`（多轮对话防漂移）

### 执行口令
确认后请回复：**按定稿实现 + 方案A**（或你选择的方案）。
```

### Ask 阶段的文档同步选型（中/重档必走）

L1 模块 README 细则见 `.cursorrules` §5；`database_map` 细则见 §13。选型决策树：

1. 只影响一个页面/模块的交互或接口？→ **L1 模块 README**（必做，见 .cursorrules §5）
2. 新增/改了表、字段、约束？→ **L3** `docs/sql/database_map.md`（写前 `get_table_schema`）
3. BOM 成本/用量/配件/纸格/导入/打印等 BOM 域规则？→ **L2** `docs/bom/BOM_RULES.md`；易回归坑 → `BOM_RISK_LIST.md`；验收场景 → `BOM_TEST_CASES.md`；页面行为 → `src/views/inv/bom/README.md`
4. 跨模块业务概念、门禁、状态机、用户可见规则话术？→ **L2** `CONTEXT.md` 对应章节
5. 难回头、有取舍、半年后还会问「为什么」？→ **L3** `docs/adr/` 新增 ADR
6. 只改 Agent 协作方式、触发词、两阶段流程？→ `.agents/skills/**`、`docs/agents/**`
7. 项目硬约束（SQL2008、pass/del、重启 API、分页）？→ `.cursorrules`（不写进 CONTEXT.md）
8. 以上都不沾且用户未要求长期沉淀？→ 定稿写「本次不同步 L2/L3」；仍可能需 L1 README

**同步优先级 tie-break**：业务概念 → CONTEXT.md；表/字段 → database_map.md；BOM 规则 → BOM_RULES.md；页面交互 → 模块 README；协作方式 → Skill/AGENTS.md；难回头取舍 → ADR。

**写文档前阅读顺序**：见 [`docs/agents/domain.md`](../../docs/agents/domain.md)。

## C. Execute 阶段规则

### 执行纪律

- **写码纪律**：遵循 [`docs/agents/coding-discipline.md`](../../docs/agents/coding-discipline.md) §1–4 及本项目 override；本 Skill 不重复展开。
- **定稿范围**：只做**最近一版、无待拍板项**的定稿中包含的内容。
- **增量定稿**：用户说「按定稿实现，另外再加 XXX」或执行中发现与定稿冲突 → **暂停**，先输出「增量定稿（待确认）」；确认后再继续，禁止边改边扩范围。
- **定稿漏了关键风险** → 暂停并问用户，不要硬做。
- **完成后**按验收标准简短打勾说明。

### 后端 API 重启

单源见 [`.cursorrules`](../../.cursorrules) §14：改了 `server/**` 或 `apiPermissionGate.js` 时收尾提醒用户手动重启并重新登录；只改前端/文档/Skill 时说明「无需重启后端 API」。

### 文档同步执行

定稿勾选了任一 L1/L2/L3 项时，Execute 必须完成：

- 先读目标文件已有章节，避免重复段落
- 只改定稿列出的文件；发现还需改其它长期文件 → 暂停，Ask 补一句定稿
- 写法：业务结果 + 边界 + 「不做什么」；表结构变更写清字段名与含义（中文）
- 冲突：代码与文档不一致 → 以本次定稿+代码为准更新文档，完成说明里点一句「已修正文档与旧描述不一致」

### 完成回复格式

中/重档完成回复须含（轻档可省略文档块）：

```
### 文档同步结果
- [x] L1 `src/views/.../README.md`：...
- [x] L2 `CONTEXT.md` §...：...
- [ ] L3 本次无
```

## 附录 D：按需查阅

### D1. 文档分层（先分层，再选文件）

| 层级 | 放什么 | 典型路径 | 生命周期 |
|------|--------|----------|----------|
| L0 临时 | 需求草稿、issue、PRD、当次验收、**重档定稿摘要** | `.scratch/<feature>/` | 功能做完可归档；不替代长期规则 |
| L1 模块 | 页面交互、接口一览、本模块坑点 | `src/views/<模块>/README.md` | 随模块演进；.cursorrules §5 强制维护 |
| L2 领域 | 跨模块业务话术、BOM/订单等已定规则 | `CONTEXT.md`、`docs/bom/*.md` | 用户说「定稿」「以后都这样」才写 |
| L3 数据/架构 | 表字段、关系、难回头技术取舍 | `docs/sql/database_map.md`、`docs/adr/` | 改表/改架构时写 |

**L0 → L2/L3 晋升**：用户说「这是长期规则 / 以后固定」，或 issue 关闭且规则已稳定时，定稿必须列出：从 .scratch 摘抄到哪些长期文件、各文件写什么；若 .scratch 有过时描述，注明是否删除或改为「已迁入 CONTEXT/…」。

**优先建议同步文档的情况**：用户说「这个规则以后固定」；产生了新业务术语或模块边界；代码行为和现有文档冲突；修复了容易再次踩坑的问题；新增了表字段/权限规则/打印规则/BOM 运算规则；新增或修改了项目协作 Skill。

### D2. 各文件写什么 / 不写什么

| 文件 | 写 | 不写 |
|------|----|------|
| `CONTEXT.md` | 已定稿业务术语、模块边界、门禁、用户可见规则 | 函数名、接口路径大全、临时 TODO、未拍板方案 |
| `docs/sql/database_map.md` | 表/字段/含义/关系/页面映射/2008 限制/历史坑 | 大段 Vue 交互说明 |
| `docs/bom/BOM_RULES.md` | BOM 领域长期规则 | 单次 bug 修复过程 |
| 模块 `README.md` | 本页按钮、弹窗、接口列表、已知问题 | 复制整份 database_map |
| `.scratch/**` | 当次 PRD、issue、验收草稿、重档定稿 | 与 CONTEXT 重复的「最终规则」 |
| `docs/adr/*` | 架构取舍（背景/决定/后果） | 日常小修小补 |

### D3. ERP 定稿检查项

涉及以下主题时，定稿里写清楚；不涉及可省略：

| 检查项 | 要写清楚什么 |
|--------|-------------|
| 影响面 | 只动哪些页面、接口、表；其它功能不动 |
| 权限门禁 | 新/改 `/api/*` 是否登记 `action_map.js`、`apiPermissionGate.js` |
| 数据安全 | 删除/覆盖是否精确到编码；不要用模糊条件扫全量 |
| 多色/多主 BOM | 是否按解析出的颜色列表处理，而不是整款通杀 |
| 数值精度 | 小数位规则；空值写 NULL 还是 0 |
| pass / 审计 | pass 默认值；操作人字段来源 |
| BOM 运算 | 是否只动 bom_cost；树形用量表界面是否保持不变 |
| 纸格导入 | 解析值和提交值如何取舍；基码和全码如何区分 |
| 性能 | 是否避免逐条查库；是否批量查询 |
| 库表文档 | 改 database_map 前 `get_table_schema` 与真表一致 |
| 后端生效 | 见 `.cursorrules` §14 |
| 测试 | 跑哪些 `node --test`、构建或手工验收 |
| 界面验收 | 有 UI 改动时是否用 IronBee 点主路径 |

### D4. 与其它 Skill 配合

- **browser-native-page**：定稿必须包含「原生新页 / 独立路由 / 无左侧栏 / 权限沿用原页面」等边界
- **diagnose**：用户报告 bug 时可结合；**重档**仍先定稿再执行，轻档可先复现再小改
- **tdd**：先用本 Skill 定稿，再进入 tdd 的红绿重构流程
- **grill-with-docs**：按其规则更积极追问术语和边界；业务术语定下来后可讨论中同步 CONTEXT.md，但仍遵守「各文件写什么/不写什么」约束
