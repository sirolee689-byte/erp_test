# Ask 定稿 -> Execute 执行

## 核心规则

在本项目里，先把需求说清楚，再动代码。

1. Ask 阶段：读代码和文档、澄清需求、对齐影响范围，最后输出「定稿」块。
2. Execute 阶段：只有用户明确授权执行后，才改代码、写 patch、跑会改变环境的命令。
3. 未获授权时：不要改代码；不要跑会写库、改文件、启动/停止服务等会改变环境的命令。
4. 如果 Codex 当前是默认可执行模式，但用户语义是“先讨论”“别动代码”“先定稿”，按本 skill 进入 Ask 阶段。

## 执行授权

以下算已经授权执行：

- `按定稿实现`
- `按定稿实现...`，后缀是执行范围，越具体越优先
- `按上面定稿实现`
- `就按定稿来`
- 对上一版完整「定稿」块回复 `同意`、`OK`、`可以改了`

以下不算授权执行：

- 只说 `定稿`
- `先说说`
- `逻辑是什么`
- `有没有疑问`
- `别动代码`
- 定稿里还有未解决的 `[ ] 待拍板`

执行时严格按“最近一份完整定稿 + 用户授权句里的范围”。如果授权句和定稿冲突，先问一句，不要猜。

## 沟通方式

默认全中文，先说结论，再说原因，最后说改哪里。

- 面向业务和界面说话，不要让函数名、变量名、文件名当主语。
- 技术词躲不开时，用一句中文解释清楚。
- 多用“改前 / 改后”、操作步骤、表格说明边界。
- 需要提代码位置时，把路径放在句末括号里。
- 不写大段代码原理，除非用户明确要求。

## Ask 阶段

可以读代码、查文档、跑只读命令来确认现状。不要写 patch。

每次需要用户确认时，输出这个模板。范围、验收、不做什么必须写清楚：

```markdown
## 定稿（待你确认）

### 目标
（一句话说明用户最终会看到什么变化）

### 范围
- 做：...
- 不做：...

### 规则与边界
1. ...

### 待你拍板（若有）
- [ ] A ... / B ...

### 验收标准
- [ ] ...

### 执行口令
确认后请回复：**按定稿实现** 或 **按定稿实现（具体范围）**
```

## Execute 阶段

- 只做定稿里的内容。
- 保持最小 diff，不顺手重构，不扩大范围。
- 匹配项目现有写法。
- 完成后按定稿里的验收标准简要打勾说明。
- Codex 执行后端改动的收尾规则：如果本次改了 `server/**` 或 `server/apiPermissionGate.js`，Codex 必须自动完成后端 API 重启，不要让用户再手动输入重启指令。
  1. 在项目根目录按 `.cursorrules` §17 重启 3001 API：释放旧 Node 进程后执行 `npm run dev:server`。
  2. 重启后必须等待并确认 `[启动指纹] bootAt=...`、同等指纹行，或 3001 新 PID。
  3. 最终回复必须贴出启动指纹或新 PID，并提醒用户后端重启后需要重新登录。
  4. 如果当前环境确实无法自动重启，必须明确说“后端还没重启”，不要断言后端改动已经生效。

## ERP 定稿检查项

涉及下列主题时，在定稿或执行前写清楚；不涉及就写 N/A 或略过。

| 检查项 | 需要写明 |
| --- | --- |
| 影响面 | 只动哪些页面、接口、表；其他功能不动 |
| 数据安全 | 删除/覆盖是否精确到编码；不要用模糊条件扫全量 |
| 多色/多主 BOM | 是否按解析出的颜色列表处理，而不是整款通杀 |
| 数值精度 | 小数位规则；空值写 `NULL` 还是 0 |
| pass / 审计 | `pass` 默认值；操作人字段来自登录用户还是固定账号 |
| BOM 运算 | 是否只动 `bom_cost`；树形用量表界面是否保持不变 |
| 纸格导入 | 解析值和提交值如何取舍；基码和全码如何区分 |
| 性能 | 是否避免逐条查库；是否批量查询 |
| 后端生效 | 是否需要 `npm run dev:server`；是否需要重新登录 |
| 测试 | 是否跑相关 `node --test` 测试 |

## 与其他项目 skill 配合

- 方案还不清楚、需要追问时，可叠加 `grill-me`。
- 定稿要同步到文档时，可叠加 `grill-with-docs`。
- 用户要求 TDD 时，先用本 skill 定稿，再按定稿进入 `tdd` 的红绿重构流程。

## 反模式

- 用户问“现在怎么实现”就直接改代码。
- 定稿还有待拍板事项就执行。
- 用猜测代替业务确认。
- Execute 时扩大范围、顺手优化或重构。
- 把单独一句“定稿”当成执行授权。
