# ask-then-execute-sale

本 Skill 只服务「销售订单模块」。目标是：**先锁定范围、先定稿、再执行**，避免全仓扫描导致额度消耗过快。

---

## A. 核心规则与口吻

### 两阶段定义

**Ask 阶段（默认）**：只读、只分析、只澄清。  
不改代码、不改文档、不启停服务、不跑会写库命令。

**Execute 阶段**：仅在用户明确授权后执行（见“执行口令”）。

### 固定分析范围（强约束）

默认只允许读取以下范围：

- 前端：`src/views/supply-chain/daily/sales-order/**`
- 后端：`server/**` 中与 `sales-order` 直接相关文件
- 测试：`test:sales-order` 涉及的测试文件
- 文档：  
  - `src/views/supply-chain/daily/sales-order/README.md`  
  - `docs/sql/database_map.md`（仅销售订单相关段落）

### 范围外处理规则

- 范围外一律不读、不分析、不建议改动。
- 如确需跨范围，必须先输出：
  1) 为什么必须跨；  
  2) 需要新增读取的文件清单（最多 5 个）；  
  3) 不跨会有什么风险；  
  然后等待用户确认。

### 先给读取清单（控额度）

在 Ask 阶段，先输出“将读取的文件清单（最多 10 个）”，等待用户确认后再深入分析。  
禁止先全仓扫描再汇报。

### 沟通口吻

- 全中文、大白话。
- 先说结论，再说原因，最后说改哪里。
- 面向业务结果，不堆术语。

## B. Ask 阶段输出模板（销售订单专用）

用户让“给方案”时，默认给 2 个方案并推荐 1 个。

## 定稿（待你确认）

### 先说结论
我推荐：方案 X。  
原因：...

### 方案选择
#### 方案 A
- 效果：...
- 优点：...
- 缺点/代价：...
- 适合：...

#### 方案 B
- 效果：...
- 优点：...
- 缺点/代价：...
- 适合：...

### 目标
（销售订单用户最终可见变化，一句话）

### 范围
- 做：仅限 sales-order 模块
- 不做：采购/外协/BOM 非必要联动逻辑

### 规则与边界
1. 不跨模块扫描。  
2. 只跑销售订单相关测试。  
3. 涉及跨模块依赖时先申请扩范围。

### 待你拍板
- [ ] 选择方案 A / B
- [ ] 是否允许跨模块（如需要）

### 验收标准
- [ ] 销售订单主链路可用（列表/详情/保存/生命周期）
- [ ] 仅执行 sales-order 相关测试
- [ ] 无无关模块改动

### 文档同步
- [ ] L1 `src/views/supply-chain/daily/sales-order/README.md`
- [ ] L3 `docs/sql/database_map.md`（仅涉及字段/表变化时）
- [ ] 本次不同步 L2/L3（如不涉及）

### 执行口令
确认后请回复：**按定稿实现 + 方案A（或B）**

---

## C. Execute 阶段规则

### 执行授权清单（仅以下算授权）

- 按定稿实现 / 按定稿实现 + 方案A / 方案B
- 就按定稿来 / 可以改了 / 执行 / 开始改

其它表达默认不执行。

### 执行纪律

- 仅改最近一版完整定稿内的内容。
- 保持最小 diff，不顺手重构，不扩大范围。
- 若发现定稿缺关键边界，暂停并回 Ask 阶段提问。

### 测试纪律（销售订单专用）

- 默认只跑 `test:sales-order` 对应测试，不跑全仓测试。
- 若用户明确要求全量测试，再扩展执行。

### 后端重启规则

若改动 `server/**` 或 `server/apiPermissionGate.js`：收尾 **一句** 提醒用户按 `.cursorrules` §16 手动 `taskkill` → `npm run dev:server` 并重新登录；**禁止** Agent 自动重启或断言「后端已生效」。  
若仅改前端/文档，明确说明「不需要重启后端 API」。

### 完成回复格式

### 文档同步结果
- [x] L1 `src/views/supply-chain/daily/sales-order/README.md`：...
- [ ] L2 本次无
- [x] L3 `docs/sql/database_map.md`：...（若涉及）
