# 入库单模块 PRD

triage: ready-for-agent

## Problem Statement

仓库、采购、外协、生产和销售退货等业务都需要把物料正式入库。旧系统已经有完整的入库单逻辑，但规则分散在旧 ASP 页面、弹窗、权限字段和库存统计口径里，容易在重做新系统时丢掉关键约束。

新系统需要建设一版可落地的“入库单”模块。它要保留旧系统的核心业务口径：入库单是库存增加的合法凭证，保存只是生成待审核单据，只有审核通过后才进入库存统计。同时，新系统不能照搬旧系统中过时的页面形态，例如独立恢复页、IE 打印兼容、前端 DOM 导出 Excel、硬编码特殊供应商等。

第一版目标是先做入库单自身闭环：用户能录入、查询、编辑、审核、反审核、删除、恢复、查看、打印入库单，并能按不同入库类型从上游单据批量带出明细。上游单据第一版只做选择、带出和保存校验，不做复杂反写。

## Solution

建设“入库单”模块，支持采购入库、外协入库、外协退料、生产入库、生产退料、销售退货、其他入库和盘盈入库等业务类型。

用户新增入库单时，先选择入库类型、仓库、关联方和关联单据，再通过批量添加或手工选料录入明细。系统在保存时由后端生成最终入库单号，并按权限决定是否自动审核。未审核单据不影响库存，审核通过后才进入库存实时汇总口径。

页面形态参考外协订单：列表为主，新增和编辑在当前模块内嵌整页表单完成，查看用只读大弹窗，打印打开独立打印页。回收站作为当前模块内的筛选或切换，不照搬旧系统独立恢复页。

## User Stories

1. As a 仓管员, I want to create a purchase inbound receipt from a purchase order, so that received supplier goods can become formal stock after approval.
2. As a 仓管员, I want to create other inbound receipts without upstream orders, so that occasional manual inbound business can be recorded.
3. As a 仓管员, I want to create inventory surplus inbound receipts without upstream orders, so that stocktake surplus can be recorded.
4. As an 外协管理员, I want to create outsourcing inbound receipts from outsourcing orders, so that returned outsourced goods can be received into warehouse.
5. As an 外协管理员, I want to create outsourcing return-material receipts from outsourcing orders, so that returned materials can be recorded correctly.
6. As a 生产主管, I want to create production inbound receipts from dispatch orders, so that finished production goods can be received into warehouse.
7. As a 生产主管, I want production inbound to require a dispatch order, so that finished-goods inbound has a clear production source.
8. As a 生产主管, I want production return to allow empty dispatch order, so that material returns without a dispatch order can still be recorded.
9. As a 销售售后人员, I want to create sales return inbound receipts from sales orders, so that returned customer goods can enter stock again.
10. As a data-entry user, I want the page to show only fields relevant to the selected inbound type, so that I do not fill unrelated supplier, customer, workshop, or order fields.
11. As a data-entry user, I want changing inbound type to clear incompatible details with a clear warning, so that wrong source materials are not mixed into the receipt.
12. As a data-entry user, I want the system to show a suggested receipt number before save, so that I can see roughly what number will be used.
13. As a data-entry user, I want the final receipt number to be generated by the server on save, so that two users saving at the same time do not need to redo their work.
14. As a data-entry user, I want the receipt number to be non-editable, so that business numbering stays controlled by the system.
15. As a data-entry user, I want external delivery, PI, PO, or paper numbers to be entered separately, so that outside document numbers do not conflict with system receipt numbers.
16. As a data-entry user, I want inbound date to default to current time, so that most receipts can be created quickly.
17. As a data-entry user, I want pending inbound date to be editable, so that I can correct the business occurrence date before approval.
18. As a data-entry user, I want approved and reviewed inbound dates to be read-only, so that formal records do not change silently.
19. As a data-entry user, I want to choose a warehouse from warehouse master data, so that stock is counted into a valid warehouse.
20. As a data-entry user, I want supplier, outsourcing customer, workshop, customer, and warehouse names to be refreshed by the server from master data, so that saved snapshots are accurate.
21. As a data-entry user, I want other inbound and surplus inbound related-party names to allow free text, so that one-off counterparties can be recorded.
22. As a data-entry user, I want batch add to import details from purchase, outsourcing, dispatch, or sales orders, so that I do not manually retype source material rows.
23. As a data-entry user, I want linked inbound types to reject source-less manual material rows, so that receipt details always trace back to the related order.
24. As a data-entry user, I want other inbound and surplus inbound to allow manual material selection, so that receipts without source orders can still be entered.
25. As a data-entry user, I want production return without dispatch order to allow manual or warehouse-filtered material selection, so that return materials can still be recorded.
26. As a data-entry user, I want each linked detail row to show available inbound quantity, so that I know the maximum quantity I can receive.
27. As a data-entry user, I want the system to block over-receipt by default, so that users cannot receive more than the available source quantity.
28. As a data-entry user, I want non-linked inbound details to require quantity greater than zero, so that meaningless zero rows cannot be saved.
29. As a data-entry user, I want no-tax mode to force tax point to zero, so that tax and amount calculations stay consistent.
30. As a price-authorized user, I want to see and edit unit price, tax, and amount fields, so that I can maintain receipt cost information.
31. As a user without price permission, I want price fields hidden in page, print, and future export, so that sensitive price information is protected.
32. As a user without price permission, I want the server to ignore or reject my price field changes, so that hidden fields cannot be overwritten through requests.
33. As a data-entry user, I want amount fields to calculate automatically from quantity, unit price, and tax, so that totals are reliable.
34. As a data-entry user, I want editing including-tax unit price to reverse-calculate excluding-tax unit price, so that both price views stay aligned.
35. As a data-entry user, I want the receipt to save as pending when auto-approval is not enabled, so that an approver can review it before stock changes.
36. As a data-entry user, I want the receipt to save as approved when my existing auto-approval config allows it, so that trusted users can complete receipt faster.
37. As an approver, I want to approve a pending receipt, so that it formally increases stock.
38. As an approver, I want to reverse-audit an approved receipt, so that wrong approved receipts can return to pending before correction.
39. As an approver, I do not want an audit rejection workflow in first version, so that the module stays simple and matches confirmed scope.
40. As an approver, I want approved receipts to be protected from direct deletion, so that stock-impacting records are not removed without reverse audit.
41. As a normal user, I want pending receipts to be editable before review, so that data-entry mistakes can be corrected.
42. As a normal user, I want approved receipts to be view-only, so that formal stock records are not changed directly.
43. As a normal user, I want reviewed `sp_flag=1` receipts to be view-only, so that financially reviewed data stays locked.
44. As a normal user, I want closed historical receipts to be view-only, so that old closed data remains compatible but protected.
45. As a normal user, I want type `8` historical processing inbound records to be read-only with an unavailable marker, so that old data can be inspected without reopening hidden business.
46. As a normal user, I want soft-deleted pending receipts to appear in recycle bin, so that accidental deletion can be restored.
47. As a normal user with delete permission, I want to restore pending deleted receipts, so that valid records can return to normal list.
48. As a super administrator, I want to permanently delete pending deleted receipts, so that test or wrong records can be fully cleaned when necessary.
49. As a viewer, I want to open a read-only detail modal, so that I can inspect header and detail data without entering edit mode.
50. As a viewer, I want to print a receipt with header, details, and totals, so that warehouse paperwork can be used offline.
51. As a viewer without price permission, I want printed receipts to hide price and amount fields, so that printed documents follow the same permission rule.
52. As an operations manager, I want operation logs for add, edit, delete, restore, audit, reverse audit, permanent delete, and auto-approval, so that every key action is traceable.
53. As a report user, I want approved and undeleted details to enter inventory statistics, so that stock quantity reflects real approved inbound.
54. As a report user, I want pending, deleted, and reverse-audited receipts excluded from stock, so that unapproved or removed records do not affect stock.
55. As a developer, I want high-risk old fields such as `kcao02`, `kcao031`, and `Describe` mapped correctly, so that old data contracts are not broken.

## Functional Requirements

### List, Search, And Recycle Bin

- The list page is the main entry for the module.
- Normal list shows non-deleted receipts.
- The page supports filters for receipt number, inbound type, warehouse, related party, related order number, audit status, inbound date range, paper number, creator, and remark keyword where practical.
- The recycle bin is a switch or filter inside the current module.
- Recycle bin shows soft-deleted pending receipts.
- Type `8`, reviewed, closed, or otherwise locked historical records can be displayed read-only.
- Export button is shown only as a pending-development placeholder in first version.
- “超量入库配置” needs a visible pending-development placeholder for future configurable over-receipt rules.

### Create

- Create form includes inbound type, inbound number suggestion, inbound date, warehouse, related party, related order number, tax mode, paper number, remark, and detail table.
- Final `kcan01` is generated by the server on save.
- The add page may display only a suggested number before save.
- `kcan01` is not manually editable.
- The number date part uses the save day, not the user-filled inbound date.
- `kcan08` stores outside paper, delivery, PI, or PO number.
- New receipt default fields include `del=0`, `closed=0`, and `sp_flag=0`.
- If auto-approval config is not satisfied, create saves `pass=0`.
- If auto-approval config is satisfied, create saves `pass=1`, writes the current saver as approver, writes approval time as save success time, and immediately enters inventory statistics.

### Edit

- Only pending, non-deleted, unreviewed, unclosed records can be edited.
- Approved records must be reverse-audited before editing.
- Reviewed `sp_flag=1` records cannot be edited.
- Closed `closed=1` records cannot be edited.
- Type `8` records cannot be edited.
- Editing saves details by whole-batch rewrite: remove old editable details and insert current page details.
- First version does not do line-level diff update.

### View

- View is read-only for every status.
- View should show header data, detail rows, audit/delete metadata where useful, and operation constraints.
- Reviewed, closed, and type `8` records should clearly show they are not operable.
- Type `8` shows a disabled or unavailable marker, such as a no-entry icon on hover.

### Print

- Print is included in first version.
- Print opens a separate print page from view.
- Print includes receipt number, inbound date, type, warehouse, related party, handler, paper number, remark, details, and totals.
- Price fields in print are controlled by price permission.
- First version uses modern print styling and does not support old IE registry-based header/footer compatibility.

### Audit And Reverse Audit

- First version supports only pending and approved: `pass=0` and `pass=1`.
- First version does not implement audit rejection.
- Audit changes header and details to approved.
- Audit writes approver and approval time.
- Approved and undeleted details enter stock statistics.
- Reverse audit changes header and details back to pending.
- Reverse-audited receipts no longer enter stock statistics.
- Reviewed, closed, deleted, and type `8` records cannot be audited or reverse-audited.
- Approved receipts cannot be deleted directly and must be reverse-audited first.

### Delete, Restore, And Permanent Delete

- Normal delete is soft delete.
- Only pending, non-reviewed, non-closed, non-type-8 records can be soft-deleted.
- Soft delete writes delete actor and delete time.
- Restore is available in recycle bin for soft-deleted pending receipts.
- Permanent delete is available only to super administrators.
- Permanent delete is limited to pending deleted records.
- Approved records must be reverse-audited before deletion or permanent deletion.

### Detail Entry

- Linked inbound types use batch add from source orders.
- Purchase inbound, outsourcing inbound, outsourcing return, production inbound, and sales return details must come from linked source documents.
- Other inbound and surplus inbound allow manual material selection.
- Production return allows no dispatch order; without dispatch order, it may filter by warehouse or manually select material.
- Detail rows must preserve material snapshot fields from material master.
- The server re-queries material master by material code when saving.
- Business input fields such as quantity, price, remark, and related-order detail key must not be overwritten by material master lookup.
- The database field `Describe` maps from frontend remark/info.
- `kcao02` stores the related source-order detail `systemcode`, not the material code.
- `kcao03` is actual stock-counting quantity.
- `kcao031` stores original inbound quantity and equals `kcao03` at initial save.

### Quantity Rules

- Linked inbound details cannot exceed available inbound quantity by default.
- First version does not implement supplier-specific over-receipt exemption.
- Future over-receipt exemption must be configurable, not hard-coded by supplier name.
- Other inbound, surplus inbound, and production return without dispatch order require quantity greater than zero.
- Quantity cap validation must be enforced on the server, not only on the page.
- No-tax mode `in_tax=2` requires tax point `0`.

### Amount Rules

- First version uses a unified amount calculation direction.
- Main inputs are excluding-tax unit price `kcao04` and tax point `tax`.
- Including-tax unit price `kcao041` is calculated from `kcao04` and `tax`.
- Excluding-tax amount `kcao05` is quantity multiplied by `kcao04`.
- Including-tax amount `kcao051` is quantity multiplied by `kcao041`.
- If the user directly edits including-tax unit price, the page reverse-calculates excluding-tax unit price.
- Price, tax, and amount fields are hidden when the user lacks price permission.
- The server must prevent users without price permission from overriding price fields.

## Inbound Type Rules

- `0` 其他入库: no source order, free-text related party, manual material details.
- `1` 采购入库: source is purchase order, supplier must match, details come from purchase order.
- `2` 外协入库: source is outsourcing order, outsourcing customer must match, details come from outsourcing order.
- `3` 外协退料: source is outsourcing order, outsourcing customer must match, details come from outsourcing order.
- `4` 生产入库: source is dispatch order, production workshop must match, dispatch order is required.
- `5` 生产退料: dispatch order may be empty; when empty, details may be selected by warehouse/manual material rules.
- `6` 销售退货: source is sales order, customer must match, details come from sales order.
- `7` 盘盈入库: no source order, free-text related party, manual material details.
- `8` 加工入库: hidden historical type; first version does not allow add or edit, only read-only display.

## Data And Field Decisions

### Header Table

Header table: `UB_ERP_Stocks_Storage`.

- `systemcode`: generated unique internal code.
- `kcan01`: receipt number generated by server on save.
- `kcan02`: inbound business date.
- `kcan03`: inbound type.
- `kcan04`: related source order number.
- `kcan05`: related party code when the type uses coded master data.
- `kcan06`: warehouse code.
- `ck`: warehouse name snapshot.
- `kcan07`: handler name.
- `kcan08`: outside paper, delivery, PI, or PO number.
- `kehu`: related party name snapshot.
- `in_tax`: tax mode, `1` taxed and `2` no-tax.
- `remark`: header remark.
- `pass`: audit status, first version uses `0` pending and `1` approved.
- `sp_flag`: review flag; first version creates `0` and respects existing `1` as locked.
- `del`: soft delete flag.
- `closed`: close flag; first version creates `0` and respects existing `1` as locked.
- creator, edit, audit, delete, IP, and time fields are written from session and request context.

### Detail Table

Detail table: `UB_ERP_Stocks_Storage_list`.

- `kcao01`: linked receipt number.
- `kcao02`: related source-order detail `systemcode`.
- `kcan04`: related source order number snapshot.
- `kcao03`: actual inbound quantity, used for stock statistics.
- `kcao031`: original inbound quantity.
- `kcao04`: excluding-tax unit price.
- `kcao041`: including-tax unit price.
- `kcao05`: excluding-tax amount.
- `kcao051`: including-tax amount.
- `tax`: tax point.
- `reference`: factory style, PI, or reference number.
- `Describe`: detail remark, mapped from frontend info/remark.
- `kcaa01` to `kcaa35`: material snapshot fields where columns exist.
- `location`: real warehouse location; do not repeat the old bug of writing version into location.
- `version`: material version.
- `sale_price`, `cost_price`, `Customer_supply`, `Customer_Name`, `seq`, `systemcode`, and `type` follow existing table contracts.

## Inventory Decisions

- First version does not add a new stock balance table.
- Inventory statistics continue to use real-time aggregation.
- Approved and undeleted inbound detail `kcao03` contributes to stock increase.
- Pending, deleted, reverse-audited, and locked-but-unapproved records do not contribute to stock increase.
- Stock aggregation dimensions are material code, warehouse code, color, version, and location.
- Empty location is grouped as empty.
- First version does not introduce a separate batch-number concept.

## Permissions

- `view`: list, detail, and print.
- `add`: create.
- `edit`: edit pending, unreviewed, unclosed, non-type-8 records.
- `audit`: audit and reverse audit.
- `delete`: soft delete and restore.
- `price`: view and edit price, tax, and amount fields.
- `export`: reserved for future backend-generated export.
- Super administrator can operate all normal permissions.
- Permanent delete is super-administrator only.

## Implementation Decisions

- Build the module as a list-first page with embedded add/edit form, read-only detail modal, module-level recycle bin, and separate print page.
- Build a receipt-number generator that finalizes `kcan01` during save and handles concurrent users without making them redo data entry.
- Build type-driven header and detail validation so each inbound type controls related party, related order, and detail source rules.
- Build batch-add services for purchase, outsourcing, dispatch, and sales source documents.
- Keep source-order reads and validation in first version, but do not implement complex upstream received-quantity write-back.
- Build detail save around whole-batch rewrite for first version.
- Build material snapshot writing on the server by re-querying material master by material code.
- Build master-data snapshot writing on the server for warehouse and coded related parties.
- Build audit and reverse-audit services that update header and details consistently.
- Build inventory statistics using approved undeleted details rather than a new balance table.
- Build permission-aware price handling across page, print, and reserved export contract.
- Build operation logs for every key state-changing action.
- Keep SQL compatible with SQL Server 2008 R2.
- Do not add new table fields in first version unless existing tables cannot represent a confirmed requirement.

## Testing Decisions

- Good tests should verify external business behavior, not internal implementation details.
- Number generation needs tests for same-day sequence, save-day prefix, concurrent save behavior, and non-editable `kcan01`.
- Save validation needs tests for required warehouse, required related order by type, production inbound requiring dispatch order, production return allowing empty dispatch order, source-party mismatch, empty details, invalid quantity, over-receipt, and no-tax tax-point validation.
- Detail mapping needs tests for `kcao02` as source detail key, `kcao031` original quantity, `Describe` remark mapping, and corrected `location` behavior.
- Price permission needs tests for page payload, save protection, print visibility, and reserved export behavior.
- Audit and reverse audit need tests confirming header/detail status sync and inventory inclusion/exclusion.
- Delete and recycle-bin tests should cover soft delete, restore, permanent delete as super admin, and blocking direct delete of approved records.
- Locked-state tests should cover `sp_flag=1`, `closed=1`, and type `8` read-only behavior.
- Print verification should cover normal user and no-price-permission user output.
- Frontend workflow verification should cover create pending receipt, edit pending receipt, audit, reverse audit, delete, restore, view, and print.

## Out Of Scope

- Audit rejection or “审核不通过”.
- Full financial review workflow.
- New auto-approval configuration page.
- Over-receipt exemption implementation.
- Over-receipt email notification.
- Complex upstream write-back of received quantity.
- Real Excel export.
- Frontend DOM-based Excel export.
- IE-specific print compatibility.
- Close and reverse-close workflow.
- Add or edit entry for hidden type `8` processing inbound.
- New stock balance table.
- Separate batch-number management.
- Line-level diff update for detail rows.

## Further Notes

- Old recovery and search pages are reference only. New system should use current module search and recycle-bin style.
- The special old supplier over-receipt exemption must not be hard-coded in first version.
- Server-side validation is required for all key rules because page-only validation cannot protect imports, automation, or crafted requests.
- This PRD intentionally follows the confirmed new-system permission model rather than the old-system permission model.
