# BOM_RISK_LIST.md

本文件记录当前 BOM 业务中已经识别的风险点。未来 AI 修改 BOM、纸格导入、成本运算、销售订单 PI BOM 前，应先阅读本文件。

## 1. P0 高风险逻辑

### 1.1 `bom_cost` 缓存未失效导致成本错误

风险：

- `GET /api/bom/tree` 命中 `bom_cost` 后会直接读缓存，不再递归 `Bom_parts`。
- 如果配件变化后没有删除缓存，页面会继续显示旧成本用量。

可能影响：

- 成本 BOM 用量表错误。
- 销售、采购、备料判断错误。
- 用户误以为修改的单位用量没有生效。

重点排查：

- 保存 `Bom_parts` 后是否只清理当前 BOM 缓存。
- 保存下层 BOM 后是否误清理了引用它的上级 BOM 缓存。
- 清理范围是否只按当前编辑 BOM 定位，而不是按被改材料编码全库找；例如改 `PC-0089/N` 时，不能把所有含该材料的 BOM 都改成未运算。

### 1.1.1 按材料编码全库反审导致误伤

风险：

- 材料编码可能被几千个 BOM 共用。
- 如果按被改材料编码反查，会把大量无关 BOM 改成未运算。
- 如果纸格导入后向上清理，会误把 `PQ-3119B1/N` 这类上级 PQ 的旧运算结果清掉。

正确做法：

- 手工保存配件明细时，只处理用户本次进入编辑的当前 BOM。
- 纸格导入写入 BAG/TAG 等下级 BOM 时，不处理引用它们的上级 PQ。

### 1.2 CUT 倍率规则被误改

风险：

- `CUT-` 自身数量不放大下层材料。
- 但 CUT 上级普通父件倍率仍应传递到下层材料。

可能影响：

- 材料用量被放大或缩小。
- 纸格导入后的成本用量严重错误。

禁止修复方式：

- 禁止把所有节点统一改成普通父子乘法。
- 禁止在前端临时修正 CUT 用量。

### 1.3 PI BOM 运算偷读主 BOM

风险：

- 销售订单 PI BOM 是订单内可改结构。
- 一键运算必须只读 PI BOM。

可能影响：

- 用户在 PI 内调整的用量被主 BOM 覆盖。
- 已确认订单物料单被错误重算。

重点排查：

- `calculate` 是否调用主 BOM 建树。
- 保存订单是否覆盖已有 PI BOM。

### 1.4 `hidePrefixes` 配置错误导致漏料

风险：

- `hidePrefixes` 会影响 `bom_cost` 落库，不只是前端隐藏。
- 如果真实材料前缀被加入隐藏列表，该材料不会进入成本结果。

可能影响：

- 成本漏项。
- 备料漏料。

建议：

- 固定默认隐藏前缀规则。
- 将隐藏前缀纳入测试。

## 2. 字段歧义风险

### 2.1 `kcac01`

风险：

- 字段名看不出业务含义。
- 容易被误认为父 BOM 编码。

固定定义：

- `Bom_parts.kcac01` 是父 `bom_000.systemcode`。

### 2.2 `kcac02`

风险：

- 容易被误认为子件编码。

固定定义：

- `Bom_parts.kcac02` 是子 BOM `systemcode`。

### 2.3 `systemcode`

风险：

- 在 `bom_000` 中是主档稳定键。
- 在 `Bom_parts` 中可能保存子 BOM systemcode。
- 在 PI BOM 中是订单内生成的新结构键。

建议：

- 任何跨表关联必须明确“是哪张表的 systemcode”。

### 2.4 `GUID` / `dr_systemcode`

风险：

- 历史缓存判断兼容 `GUID`。
- 纸格导入会写三连键。

禁止：

- 禁止无条件用 `GUID` 替代 `systemcode`。

### 2.5 `sid`

风险：

- `bom_cost.sid` 在库存 BOM 中是主档 `systemcode`。
- `UB_ERP_Bom_pi_cost.sid` 在销售订单中是 PI 号。

禁止：

- 禁止跨表复制 `sid` 逻辑时不区分业务域。

### 2.6 `kcac05`

风险：

- 前端可能按百分比输入。
- 库内保存小数。
- 运算中可能 fallback 到 `kcaa33`。

待业务确认：

- `kcac05=0` 是明确无损耗，还是允许使用 `kcaa33`。

### 2.7 `Describe`

风险：

- 它不是普通展示备注。
- 成本展示汇总按 `kcaa01 + Describe` 合并。

可能影响：

- 同编码不同搭配被错误合并。
- 同编码同搭配被错误拆分。

### 2.8 `Seq`

风险：

- 普通配件排序、纸格 CUT 预览、Material 全局序、Accessory 偏移都使用或影响排序。

可能影响：

- 成本 BOM 用量表排序不稳定。
- 纸格导入顺序与页面顺序不一致。

## 3. N+1 查询风险

### 3.1 销售订单保存明细逐行查主档

位置：

- 销售订单保存明细快照逻辑。

风险：

- 明细多时，每行查一次 `bom_000`。

建议：

- 未来抽成批量读取主档快照。

### 3.2 PI BOM 创建逐行 INSERT

位置：

- 从主 BOM 建 PI BOM。

风险：

- 大 BOM 或多层 BOM 导致事务时间长。

建议：

- 未来抽成批量插入服务。

### 3.3 手工保存 `Bom_parts` 逐行 lookup

位置：

- 手工保存配件明细。

风险：

- 每行查子 BOM systemcode、重复行、再更新。

建议：

- 未来批量预取子件主档和已有配件行。

### 3.4 PI BOM 保存逐行 UPDATE

位置：

- 保存 PI BOM 用量。

风险：

- 大 BOM 下逐行更新慢。

建议：

- 未来用批量更新或临时表策略。

## 4. 缓存风险

### 4.1 有缓存即直读

风险：

- 缓存存在时不再递归。

要求：

- 所有结构变更必须清缓存。

### 4.2 主档一键更新不重算

风险：

- 一键更新会同步基础字段到 `bom_cost`，但不改用量。

可能影响：

- 用户误以为已经重新计算成本。

要求：

- 文案必须明确“用量未改、未重算”。

### 4.3 隐藏前缀变更不自动补缓存

风险：

- 旧缓存不包含当时被隐藏的行。
- 后续改隐藏前缀不会自动恢复这些行。

要求：

- 改隐藏前缀后若要影响落库结果，必须重新运算。

### 4.4 列清单进程缓存

风险：

- 表字段在线变更后，Node 进程内列缓存可能未刷新。

要求：

- 数据库结构变化后必须重启 API。

## 5. 可能导致库存错误的位置

### 5.1 `Bom_parts.kcac01` 写错

影响：

- 配件挂到错误父 BOM。
- 当前 BOM 查不到配件。
- 错误 BOM 多出配件。

### 5.2 `Bom_parts.kcac02` 写错

影响：

- 递归进入错误子 BOM。
- 成本用量包含错误下层材料。

### 5.3 纸格导入 CUT 编码错误

影响：

- 主 BOM 与 CUT 子 BOM 链接断裂。
- Material 无法挂到正确 CUT。

### 5.4 删除 BOM 树范围错误

影响：

- 误删其它颜色、其它款或共用物料。

要求：

- 删除纸格 BOM 树必须按明确主 BOM 或 systemcode 定位，不得靠宽泛 LIKE 猜测。

### 5.5 PI BOM 删款不同步

影响：

- 订单明细已删，但 PI BOM 仍残留。
- 后续物料单包含不在单款。

## 6. 可能导致成本错误的位置

### 6.1 损耗 fallback 口径不清

影响：

- `kcac05=0` 时可能错误使用 `kcaa33`。

建议：

- 业务确认后写死规则，并补测试。

### 6.2 `Describe` 合并键处理错误

影响：

- 同料不同搭配被合并。
- 成本汇总数量不符合纸格搭配。

### 6.3 `hidePrefixes` 漏料

影响：

- 被隐藏前缀的真实材料不进成本。

### 6.4 CUT 倍率误算

影响：

- 成本用量成倍偏差。

### 6.5 `bom_cost` 与 `Bom_consumption` 混用

影响：

- 读取历史停用表得到旧结果。

要求：

- 当前库存 BOM 成本结果以 `bom_cost` 为准。

### 6.6 订货数量误写入 PI 成本结构用量

影响：

- PI 物料单结构用量被放大。
- 展示再乘订货数量时二次放大。

要求：

- 订货数量只用于展示备料，不写入结构运算结果。

## 7. 审核状态风险

### 7.1 前端禁用不等于安全

风险：

- 用户可绕过前端请求接口。

要求：

- 后端必须按 `pass` 和 `del` 再拦截。

### 7.2 配件行 `pass` 被误用

风险：

- 纸格导入会写 `Bom_parts.pass`。
- 但库存配件编辑应由父 BOM `pass` 控制。

要求：

- 禁止用 `Bom_parts.pass` 单独判断配件能否编辑。

## 8. 排查顺序

成本或库存异常时，按以下顺序排查：

1. 当前页面是否命中旧后端进程。
2. 当前 BOM 主档 `systemcode` 是否正确。
3. `Bom_parts.kcac01` 是否挂到正确父 `systemcode`。
4. 子件如有下层，`kcac02` 是否指向正确子 `systemcode`。
5. 是否存在 `bom_cost` 旧缓存。
6. 最近一次配件保存是否清理缓存。
7. `hidePrefixes` 是否把真实材料排除。
8. CUT 倍率是否按特殊规则处理。
9. `kcac05` 与 `kcaa33` 损耗来源是否符合业务口径。
10. 销售订单场景下，是否错误读取了主 BOM 而不是 PI BOM。

## 9. 禁止修复方式

- 禁止在前端硬算最终成本并写库。
- 禁止手工 UPDATE `bom_cost` 当作重算。
- 禁止只清当前 BOM 缓存，不清上级引用缓存。
- 禁止为解决一条数据错误而改全局递归规则。
- 禁止绕过审核状态直接保存配件。
- 禁止把 `Bom_consumption` 恢复为有效表而不补测试。
- 禁止以字段名相似为理由猜测关联关系。

## 10. 建议长期改造方向

- 抽 `BomStructureService` 统一处理 `bom_000` / `Bom_parts` 结构。
- 抽 `BomUsageCalculationService` 统一处理用量、损耗、CUT 规则。
- 抽 `BomCostCacheService` 统一处理 `bom_cost` 缓存读取、失效、重算。
- 抽 `BomPartsSyncService` 统一处理主档字段同步。
- 抽 `PiBomService` 统一处理销售订单 PI BOM 建款、删款、同步。
- 抽 `MaterialBillService` 统一处理 PI 物料单生成。
- 建立 `BomFieldDictionary`，固定 `kcaa*`、`kcac*` 字段含义。
