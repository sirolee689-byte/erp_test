# Ask 定稿 -> Execute 执行

这个 Skill 是本 ERP 项目的“先对齐，再动手”流程。用户经常希望先看到几个可选方案、你推荐一个、说明边界，再由用户拍板执行。

**维护位置（单源）**：

- **唯一真源目录**：`.agents/skills/ask-then-execute/`（`SKILL.md`、`agents/openai.yaml`；Codex 桌面读本目录）
- **Cursor**：`.cursor/skills/ask-then-execute/` 为 Windows **目录联接（junction）**，指向上述目录；**只改 `.agents` 下文件，不要单独改 `.cursor` 里另一份拷贝**
- 联接丢失时（换机、克隆仓库后）：在项目根执行 `npm run link:skill:ask-then-execute`
- 历史备份：同目录 `SKILL.backup.md`（不参与加载）

## 核心规则

1. Ask 阶段只允许读代码、查文档、跑只读命令、澄清需求；不要写 patch，不要改文件，不要启停服务，不要跑会写库的命令。
2. Execute 阶段只有在用户明确授权后才开始改代码。
3. 未授权时，即使你已经知道怎么改，也只能输出“定稿（待确认）”。
4. 如果需求里有“如有疑问先问”“禁止自行猜测”“只是讨论”“先给方案”等语义，必须进入 Ask 阶段。
5. 如果用户指定了方案，例如“按定稿实现 + 方案B”，执行时只做该方案，不要混用其它方案。

## 沟通口吻

- 默认全中文，大白话。
- 先说结论，再说原因，最后说改哪里。
- 少用技术术语；必须用时，用一句话解释。
- 面向业务结果说话：用户会看到什么变化、为什么这样更稳。
- 不写大段代码原理，除非用户明确要。
- 用户让“给方案”时，不要只给一个方案；至少给 2 个方案，通常给 2-3 个即可。
- 必须明确推荐一个方案，并说明推荐理由和代价。

## Ask 阶段输出要求

Ask 阶段要给“定稿”，不要写成松散聊天。默认模板如下：

```markdown
## 定稿（待你确认）

### 先说结论
我推荐：方案 X。
原因：...

### 方案选择
#### 方案 A：...
- 效果：...
- 优点：...
- 缺点/代价：...
- 适合：...

#### 方案 B：...
- 效果：...
- 优点：...
- 缺点/代价：...
- 适合：...

（如有必要再给方案 C）

### 目标
（一句话说明用户最终会看到什么变化）

### 范围
- 做：...
- 不做：...

### 规则与边界
1. ...
2. ...

### 待你拍板（如有）
- [ ] 选择方案 A / B / C
- [ ] 其它需要用户确认的问题

### 验收标准
- [ ] ...
- [ ] ...

### 文档同步
- [ ] **L1 模块 README**（.cursorrules §5 强制）：路径 `...`；更新要点：...
- [ ] **L2 领域**：`CONTEXT.md` 章节 / `docs/bom/...`；写什么：...
- [ ] **L3 数据/架构**：`docs/sql/database_map.md` / `docs/adr/`；写什么：...
- [ ] **L0 仅临时**：只更新 `.scratch/...`；不晋升原因：...
- [ ] **本次不同步 L2/L3**，原因：...

**写文档前阅读顺序**：见 `docs/agents/domain.md` — CONTEXT 相关章 → 模块 README → `docs/sql/database_map.md` → ADR。

### 执行口令
确认后请回复：**按定稿实现 + 方案A**（或你选择的方案）。
```

### 什么时候可以少给方案

只有在用户需求非常明确、没有合理分支时，才可以只给一个“定稿方案”。但仍然要写清楚：
- 为什么不需要多方案；
- 做什么；
- 不做什么；
- 验收标准。

### 什么时候必须追问

下面情况不要猜，先问：
- 需求会影响数据删除、覆盖、批量更新、审核状态、成本运算、打印输出等关键结果。
- 用户给的例子和真实数据不一致。
- 方案有明显业务取舍，但用户没有表达偏好。
- “当前页/全部/勾选项”“覆盖/跳过”“新窗口/当前页”等边界不清楚。

追问最多 1-3 个问题，问完就停，不要顺手实现。

## 执行授权

以下算作授权执行：

- `按定稿实现`
- `按定稿实现 + 方案A`
- `按定稿实现 + 方案B`
- `按上面定稿实现`
- `就按定稿来`
- `可以改了`
- `同意`
- `OK`
- 用户明确说“执行”“开始改”“照这个改”

以下不算授权执行：

- `定稿`
- `先说说`
- `逻辑是什么`
- `有没有疑问`
- `别动代码`
- 只是在讨论方案
- 定稿里还有未解决的 `[ ] 待拍板`

如果授权句和定稿冲突，先问一句确认，不要猜。

## Execute 阶段规则

- 只做最近一版完整定稿中包含的内容。
- 保持最小 diff，不顺手重构，不扩大范围。
- 匹配项目现有写法，不为了“更优雅”改无关代码。
- 如果发现定稿漏了关键风险，暂停并问用户，不要硬做。
- 完成后按验收标准简短打勾说明。
- 如果定稿写了需要同步文档，文档更新也属于本次执行范围，不要只改代码。
- 完成回复须含 **「文档同步结果」** 小节（格式见下文「Execute 阶段：文档同步清单」）。

## 文档同步规则

用户很在意项目知识能沉淀下来。Ask 阶段须走一遍 **选型决策树**；Execute 阶段按定稿同步，且与 `.cursorrules` §5 对齐。

### 长期知识分层（先分层，再选文件）

| 层级 | 放什么 | 典型路径 | 生命周期 |
| --- | --- | --- | --- |
| L0 临时 | 需求草稿、issue、PRD、当次验收 | `.scratch/<feature>/`（含 `PRD.md`、issues） | 功能做完可归档；**不**替代长期规则 |
| L1 模块 | 页面交互、接口一览、本模块坑点 | `src/views/<模块>/README.md` | 随模块演进；**.cursorrules §5 强制维护** |
| L2 领域 | 跨模块业务话术、BOM/订单等已定规则 | `CONTEXT.md`、`docs/bom/*.md` | 用户说「定稿」「以后都这样」才写 |
| L3 数据/架构 | 表字段、关系、难回头技术取舍 | `docs/sql/database_map.md`、`docs/adr/` | 改表/改架构时写 |

**晋升规则（L0 → L2/L3）**：用户在 Ask 里说「这是长期规则 / 以后固定」，或 issue 关闭且规则已稳定时，定稿必须列出：从 `.scratch` 摘抄到哪些长期文件、各文件写什么；若 `.scratch` 有过时描述，注明是否删除或改为「已迁入 CONTEXT/…」。

### 选型决策树（Ask 阶段必走一遍）

1. **只影响一个页面/模块的交互或接口？** → 优先 **L1 模块 README**（必做，见 `.cursorrules` §5）。
2. **新增/改了表、字段、约束、与页面字段对应？** → **L3** `docs/sql/database_map.md`（不要只在 README 里写表结构当唯一来源）。
3. **BOM 成本/用量/配件/纸格/导入/打印等 BOM 域规则？** → **L2** `docs/bom/BOM_RULES.md`；易回归坑 → `BOM_RISK_LIST.md`；验收场景 → `BOM_TEST_CASES.md`；页面行为细节 → `src/views/inv/bom/README.md`。
4. **跨模块业务概念、门禁、状态机、用户可见规则话术？** → **L2** `CONTEXT.md` 对应章节（不写实现细节、不写大段 SQL）。
5. **难回头、有取舍、半年后还会问「为什么」？** → **L3** `docs/adr/` 新增 ADR（背景 / 决定 / 后果）。
6. **只改 Agent 协作方式、触发词、两阶段流程？** → `.agents/skills/**`（Cursor 通过 junction 可见）、`docs/agents/**`；入口摘要必要时动 `AGENTS.md`。
7. **项目硬约束（SQL2008、pass/del、重启 API、分页）？** → `.cursorrules` / `.cursor/rules/**`（**不要**写进 `CONTEXT.md`）。
8. **以上都不沾、且用户未要求长期沉淀？** → 定稿写「本次不同步 L2/L3」；仍可能需 **L1 README**。

决策树冲突时，按下方 **同步优先级** 取舍。

### 各文件写什么 / 不写什么（摘要）

| 文件 | 写 | 不写 |
| --- | --- | --- |
| `CONTEXT.md` | 已定稿业务术语、模块边界、门禁、用户可见规则 | 函数名、接口路径大全、临时 TODO、未拍板方案 |
| `docs/sql/database_map.md` | 表/字段/含义/关系/页面映射/2008 限制/历史坑 | 大段 Vue 交互说明 |
| `docs/bom/BOM_RULES.md` | BOM 领域长期规则 | 单次 bug 修复过程 |
| 模块 `README.md` | 本页按钮、弹窗、接口列表、已知问题 | 复制整份 database_map |
| `.scratch/**` | 当次 PRD、issue、验收草稿 | 与 CONTEXT 重复的「最终规则」 |
| `docs/adr/*` | 架构取舍 | 日常小修小补 |

### 长期知识地图（完整路径）

- `CONTEXT.md`：业务词汇、领域概念、模块边界、已经定稿的业务说法。不要写代码实现、接口细节、临时计划。
- `docs/sql/database_map.md`：数据库主文档。写表、字段、字段含义、表关系、接口/页面对应关系、关键约束、历史坑点、SQL Server 2008 R2 限制。
- `docs/database_map.md`：数据库文档入口说明，通常只指向 `docs/sql/database_map.md`，不要把正文重复写两份。
- `docs/bom/BOM_RULES.md`：BOM 长期业务规则，例如成本用量、配件明细、CUT/PI/纸格导入相关规则。
- `docs/bom/BOM_RISK_LIST.md`：BOM 风险点、历史坑、容易回归的问题。
- `docs/bom/BOM_TEST_CASES.md`：BOM 验收场景、关键测试用例。
- `src/views/inv/bom/README.md`：BOM 页面级维护说明、交互规则、接口一览、打印/导出等页面行为。
- 各模块 `README.md`：对应模块自己的长期维护说明，例如采购报价、销售订单、纸格导入、宿舍、人事、权限等。
- `.scratch/<feature>/`：PRD、issue、当次验收（L0）；稳定规则须晋升到 L2/L3。
- `docs/agents/**`：Agent 文档地图、写码纪律、领域文档说明、issue/triage 规则。
- `AGENTS.md`：Codex 入口和项目级协作规则摘要。
- `.agents/skills/**/SKILL.md`：项目专用工作流（`ask-then-execute` 经 junction 供 Cursor 读取）。
- `.cursorrules` / `.cursor/rules/**`：项目硬约束和环境规则，例如 SQL Server 2008 R2、分页、`pass/del`、后端 API 重启。
- `docs/adr/**`：只记录难回头、有真实取舍、未来别人会问“为什么这样做”的架构或技术决策。

### 同步优先级（决策树 tie-break）

1. 业务概念、模块边界、长期业务话术：优先 `CONTEXT.md`。
2. 表、字段、字段含义、表关系、数据库限制：优先 `docs/sql/database_map.md`。
3. BOM 业务规则：优先 `docs/bom/BOM_RULES.md`；风险坑 → `BOM_RISK_LIST.md`；验收例子 → `BOM_TEST_CASES.md`。
4. 某个页面自己的交互和维护说明：优先该模块 README。
5. 协作方式、Agent 行为、触发词：优先 Skill 或 `AGENTS.md`。
6. 难回头的技术/架构取舍：考虑 ADR。

### 与 `.cursorrules` §5 的关系

- **§5**：完成功能 → **必须**更新**当前目录** `README.md`（大白话、表变动要点）。
- **本 skill**：定稿「文档同步」区分 **L1（§5 强制）** 与 **L2/L3（定稿勾选）**。
- Execute：**先代码，再文档**；定稿勾选的文档未完成，不得宣称验收通过。
- 用户说「不要写文档」：定稿可勾选「本次不同步 L2/L3」；**仍默认要 L1 README**，除非用户明确「连 README 也不要动」。

### 定稿里必须说明（文档部分）

1. 这次要不要同步 L2/L3；L1 README 路径与要点。
2. 如果要，同步哪些文件、各文件写什么类型内容。
3. 若不需要 L2/L3，写清原因。
4. 是否涉及 L0→L2/L3 晋升。

### 遇到这些情况，优先建议同步文档

- 用户明确说“这个规则以后固定”“这是定稿”“这就是我要的结果”。
- 需求产生了新的业务术语、模块边界或长期规则。
- 发现代码行为和现有文档冲突。
- 修复了一个容易再次踩坑的问题。
- 新增了数据库字段、表关系、精度规则、权限规则、打印规则、BOM 运算规则。
- 新增或修改了项目协作 Skill。

### Execute 阶段：文档同步清单

定稿若勾选任一项 L1/L2/L3，Execute **必须**完成：

1. **先读**目标文件已有章节，避免重复段落。
2. **只改**定稿列出的文件；发现还需改其它长期文件 → 暂停，Ask 补一句定稿。
3. **写法**：业务结果 + 边界 +「不做什么」；表结构变更写清字段名与含义（中文）。
4. **冲突**：代码与文档不一致 → 以**本次定稿+代码**为准更新文档，完成说明里点一句「已修正文档与旧描述不一致」。
5. **完成回复**必须含：

```markdown
### 文档同步结果
- [x] L1 `src/views/.../README.md`：...
- [x] L2 `CONTEXT.md` §...：...
- [ ] L3 本次无
```

### 与 `grill-with-docs` 配合

- 按 `grill-with-docs` 更积极追问术语和边界。
- 业务术语定下来后，可讨论中同步 `CONTEXT.md`，但仍遵守「各文件写什么/不写什么」表。
- 不要把实现细节塞进 `CONTEXT.md`；实现细节放到 `docs/sql/database_map.md`、`docs/bom/**`、ADR、模块 README。

## 后端 API 重启规则

如果本次改了 `server/**` 或 `server/apiPermissionGate.js`：

1. 必须把“重启后端 API”当成本次执行的一部分，不要只提醒用户。
2. 按 `.cursorrules` §17 在项目根目录重启 3001 API：通常是释放旧 Node 进程后执行 `npm run dev:server`。
3. 等到控制台出现 `[启动指纹] bootAt=...`、同等指纹行，或确认 3001 新 PID。
4. 最终回复里贴出启动指纹或新 PID，并提醒用户后端重启后需要重新登录。
5. 如果环境限制导致无法自动重启，必须明确说：**后端还没重启**。不要断言后端改动已经生效。

如果只改前端、文档、Skill 文件，最终说明“不需要重启后端 API”。

## ERP 定稿检查项

涉及下面主题时，定稿里要写清楚；不涉及可以省略：

| 检查项 | 要写清楚什么 |
| --- | --- |
| 影响面 | 只动哪些页面、接口、表；其它功能不动 |
| 数据安全 | 删除/覆盖是否精确到编码；不要用模糊条件扫全量 |
| 多色/多主 BOM | 是否按解析出的颜色列表处理，而不是整款通杀 |
| 数值精度 | 小数位规则；空值写 `NULL` 还是 0 |
| pass / 审计 | `pass` 默认值；操作人字段来源 |
| BOM 运算 | 是否只动 `bom_cost`；树形用量表界面是否保持不变 |
| 纸格导入 | 解析值和提交值如何取舍；基码和全码如何区分 |
| 性能 | 是否避免逐条查库；是否批量查询 |
| 后端生效 | 是否需要 `npm run dev:server`；是否需要重新登录 |
| 测试 | 跑哪些 `node --test`、构建或手工验收 |

## 与其它 Skill 配合

- 如果用户同时提到 `browser-native-page`，定稿必须包含“原生新页 / 独立路由 / 无左侧栏 / 权限沿用原页面”等边界。
- 如果用户报告 bug，可结合 `diagnose`，但仍遵守“先定稿，后执行”。
- 如果用户要求 TDD，先用本 Skill 定稿，再进入 `tdd` 的红绿重构流程。

## 反模式

- 用户只是让讨论，就直接改代码。
- 只给一个方案，却不说明为什么没有其它方案。
- 推荐方案但不说明代价。
- 定稿没有写“不做什么”。
- 定稿还有待拍板事项就执行。
- Execute 阶段顺手扩大范围、重构、改样式、改文案。
- 改了后端却不自动重启 API，也不说明后端是否已生效。
- 定稿未列文档目标却擅自大改 `CONTEXT.md`。
- 该写 `database_map` 却只改了 README。
- 文档同步未完成就勾选验收标准。
- 把接口实现、SQL 全文塞进 `CONTEXT.md`。
- 在多个文件重复粘贴同一段表结构（database_map 为单源）。
- 用 `.scratch` 当永久规则库，从不晋升到 L2/L3。
