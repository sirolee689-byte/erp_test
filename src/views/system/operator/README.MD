# 操作员资料（`system/operator`）

### 1. 业务逻辑（白话版）
- **这个功能是干嘛的？**：管理系统的“操作员账号”（也就是能登录系统的人），可以查看列表、新增账号、编辑账号、禁用账号。
- **谁会用？**：一般是管理员或人事/IT 人员（看你们公司权限怎么分）。
- **为什么是“禁用”而不是“真删除”？**
  - 因为真删除会丢历史记录，后面不好追查
  - 所以我们用 `Status=0` 表示“禁用”（软删除）

### 2. 数据库关联
- **对应表名**：`Sys_Users`
- **本模块用到的字段（当前代码用到）**
  - `UserID`：用户 ID（主键，用来定位“改哪一行”）
  - `UserCode`：工号
  - `UserName`：姓名
  - `Password`：密码（当前按需求先直存，后续建议加密）
  - `Status`：状态（1=启用，0=禁用）
  - `CreatedAt`：创建时间
  - `RoleID`：角色外键（v1.0.7 起必填，对应 `Sys_Roles`）
- **角色下拉数据**：页面加载时请求 `GET /api/roles`，参数为 `page=1&pageSize=500&status=1`（分页接口，只取启用角色列表给 `el-select` 用）。

### 3. 已完成的功能（CRUD 进度表）
- [x] **查（列表展示+分页+搜索）**：页面表格展示 `Sys_Users`（GET `/api/users`）
- [x] **增（新增操作员）**：弹窗录入并写入数据库（POST `/api/users`）
- [x] **改（编辑操作员）**：弹窗回显旧数据并更新数据库（PUT `/api/users`）
- [x] **删（软删除=禁用）**：二次确认后把 `Status` 改为 0（PUT `/api/users`）

### 4. 模块开发标准（v1.0.4 最终版）
> 这一段是“以后开发类似模块照着做”的标准，避免你每次都从头想一遍。

#### 4.1 参数规范（前端 → 后端）
- **分页参数**（所有列表页面统一）
  - `page`：第几页（默认 1）
  - `pageSize`：每页多少条（默认 10，可选 10/20/50/100）
- **搜索参数**（所有列表页面统一）
  - `keyword`：模糊搜索关键字（本模块含义：工号或姓名）
- **视图参数**（双视图切换统一）
  - `status`：当前视图状态
    - `1`：在职操作员（启用）
    - `0`：恢复操作员（回收站/禁用）

#### 4.2 返回规范（后端 → 前端）
- **分页列表接口必须返回**
  - `code`：200 表示成功
  - `msg`：成功/失败的中文信息
  - `list`：本页数据数组
  - `total`：总条数（给 `el-pagination` 用）

#### 4.3 页面交互标准（前端）
- **标准搜索栏**
  - 位置：表格上方的独立区域
  - 交互：支持回车搜索 + “查询/重置”
  - 联动：搜索时必须把页码重置为 1
- **标准分页**
  - 表格下方 `el-pagination`
  - 默认每页 10 条，支持 `[10,20,50,100]`
- **标准 CRUD**
  - 新增：弹窗表单 → POST
  - 编辑：弹窗回显旧数据 → PUT
  - 删除：本模块用“禁用”（Status=0）代替真删除

### 5. 接口说明（你可以照着 Postman/浏览器测试）
> 小白提醒：改完后端代码后，通常需要重启后端：`npm run dev:server`

#### 5.1 查询用户列表（查 + 分页 + 搜索）
- **方法/路径**：GET `/api/users`
- **返回格式（固定）**
  - `code: 200`
  - `msg: "success"`
  - `list: 用户数组（本页）`
  - `total: 总条数（给分页器用）`
- **分页参数**
  - `page`：第几页（默认 1）
  - `pageSize`：每页多少条（默认 10）
- **搜索参数**
  - `keyword`：模糊搜索关键字（工号或姓名）
- **双视图切换规则（本模块母版逻辑）**
  - 前端用一个变量 `selectedStatus` 控制当前视图
  - 前端请求时把 `status=selectedStatus` 传给后端
  - 后端 SQL 根据 `status` 加 WHERE：
    - `status=1` → `WHERE Status = 1`（在职视图）
    - `status=0` → `WHERE Status = 0`（回收站视图）
  - 如果同时传了 `keyword`，SQL 会在当前视图的基础上再加 LIKE 模糊匹配
- **示例**
  - 不搜索：`/api/users?page=2&pageSize=10`
  - 搜索：`/api/users?page=1&pageSize=10&keyword=张`
  - 在职视图：`/api/users?status=1&page=1&pageSize=10`
  - 回收站视图：`/api/users?status=0&page=1&pageSize=10`
- **页面怎么用的（数据从数据库到网页）**
  - 页面加载 → `axios.get('/api/users', { params: { page, pageSize, keyword } })`
  - Vite 把 `/api` 代理到后端（见 `vite.config.js`）
  - 后端查询 `Sys_Users` → 把 `list/total` 返回给前端
  - 前端把 `list` 放进 `users` → `el-table` 自动渲染
  - **分页怎么工作的（超小白解释）**
    - 你点分页器 → 前端改 `page/pageSize`
    - 前端把 `page/pageSize` 放到 URL 查询参数里发给后端
    - 后端计算 `offset = (page - 1) * pageSize`
    - SQL 优先用 `OFFSET @offset ROWS FETCH NEXT @pageSize ROWS ONLY` 只查这一页（不支持时自动降级）
  - **搜索怎么工作的（超小白解释，重点讲 LIKE）**
    - 你在输入框输入关键字，比如：`张`
    - 前端把 `keyword=张` 传给后端
    - 后端把它拼成 `@key = '%张%'`
    - SQL 写成：`UserCode LIKE @key OR UserName LIKE @key`
    - `LIKE` 的意思是“像……一样”
      - `%` 的意思是“任意长度的任意字符”
      - 所以 `%张%` 就表示“只要中间包含 张 就算匹配”
  - **双视图切换怎么工作的（超小白解释）**
    - 你点“在职操作员/恢复操作员”
    - 前端把一个变量 `selectedStatus` 改成 1 或 0
    - 前端请求时带上 `status=1/0`
    - 后端拿到 status 后，SQL 就知道该查 `Status=1` 还是 `Status=0`
    - 这就是“一个变量控制 SQL 语句切换”的核心设计

#### 5.2 新增用户（增）
- **方法/路径**：POST `/api/users`
- **前端提交字段**
  - `UserCode`：工号
  - `UserName`：姓名
  - `Password`：密码
- **后端自动补的默认值**
  - `Status = 1`（启用）
  - `CreatedAt = GETDATE()`（当前时间）

#### 5.3 编辑用户（改）
- **方法/路径**：PUT `/api/users`
- **前端提交字段**
  - `UserID`：必须有（告诉后端“改哪一行”）
  - `UserCode`：工号
  - `UserName`：姓名
  - `Password`：可选（留空表示“不修改密码”）
- **“旧数据回显”是怎么做到的？（超小白解释）**
  - 你点击某一行的“编辑”
  - 前端拿到那一行 `row`（里面就是旧数据）
  - 把 `row.UserCode / row.UserName` 赋值到表单对象 `createForm`
  - 输入框用 `v-model="createForm.xxx"` 绑定了表单对象
  - 所以输入框就自动显示旧值（这就叫回显）

#### 5.4 禁用用户（删=软删除）
- **方法/路径**：PUT `/api/users`
- **前端提交字段**
  - `UserID`
  - `Status: 0`
- **前端保护**：点击“禁用”会弹二次确认框，防止误点
 - **提示语**：禁用成功后提示 **“已移入回收站”**

#### 5.5 恢复用户（把禁用账号改回启用）
- **方法/路径**：PUT `/api/users/resume`
- **前端提交字段**
  - `UserID`
- **效果**：把 `Status` 从 0 改回 1
 - **提示语**：恢复成功后提示 **“已恢复在职”**

#### 5.6 彻底删除用户（物理删除）
- **方法/路径**：DELETE `/api/users/:id`
- **阶梯式删除规则（重要）**
  - **必须先禁用（Status=0）才能删除**
  - 启用账号（Status=1）禁止直接物理删除
  - 小白版解释：先“放回收站”（禁用），再“清空回收站”（删除）
 - **二次确认**：前端会弹出“此操作不可逆”的危险警告弹窗

### 6. 已知问题 / 下一步计划（大白话）
- **工号重复校验（已做后端提示）**：
  - 如果数据库已有唯一约束，新增时出现重复会被后端识别（错误号 2627 / UNIQUE KEY）
  - 前端会直接弹出：**“工号已存在，请勿重复添加”**
  - 下一步建议：数据库层面继续保持/补齐唯一索引（最可靠）
- **密码加密**：目前密码按需求先直存，后续必须改成“加盐哈希”（更安全）。
