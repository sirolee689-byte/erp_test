# 登录模块（`views/login`）v1.0.6

### 1. 这个功能是干嘛的？（大白话）
- **目的**：让“操作员账号”（`Sys_Users`）可以登录 ERP 后台页面。
- **效果**：
  - 你没有登录就直接访问后台地址（比如 `/system/operator`）会被拦截并跳回登录页。
  - 你登录成功后，会自动进入后台首页，或者回到你刚才想去的页面。

### 2. 数据库关联（你最关心的表）
- **表名**：`Sys_Users`
- **旧系统物理表（兼容）**：若存在列 `uid`（主键，对应 `Hr_staff.id`）、`username`（登录账号）、`usercode`（账号编码）、`password`、`is_admin` 等，且无 `UserID` 列，则后端按「旧表布局」识别：`is_admin=1` 视为超级管理员（菜单全通）；登录匹配 `username` 或 `usercode`；人事档案姓名通过 `uid` 左联 `Hr_staff`。
- **本登录用到的字段（ERP 标准表）**
  - `UserName`：登录框按本列匹配
  - `UserCode`：账号编码（审计 token 中的工号语义；登录框不按此列匹配 ERP 表）
  - `Password`：密码（v1.0.6 起：bcrypt 加密字符串；兼容旧明文测试数据）
  - `Status`：状态（`1=启用`，`0=禁用`）

### 3. 后端接口：`POST /api/login`
- **位置**：`server/index.js`
- **请求体（前端发给后端）**
  - `Account`：登录输入（后端用其仅匹配 `UserName`）
  - `Password`：密码
- **校验顺序（按需求写死的）**
  1. **账号是否存在**：`UserName` 不匹配则返回 `账号不存在`
  2. **账号是否禁用**：`Status=0` 就返回 `账号已被禁用，请联系管理员`
  3. **密码是否正确**：不匹配就返回 `密码错误`
- **成功返回（统一格式）**
  - `code: 200`
  - `msg: "success"`
  - `data.token`：后端生成的随机字符串（登录凭证）
  - `data.user`：用户信息（不包含密码）

### 4. 前端登录页：`src/views/login/index.vue`
- **页面包含**
  - 背景（渐变风格，简约大气）
  - 用户名输入框（对应 `UserName`）
  - 密码输入框
  - 登录按钮
- **登录成功后做了什么？**
  - 把后端返回的 `token` 存到浏览器：`localStorage.setItem('erp_token', token)`
  - 把用户信息（可选）存到浏览器：`localStorage.setItem('erp_user', JSON.stringify(user))`
  - 跳转到：
    - 优先回到 `?redirect=xxx` 指定的页面
    - 否则回到 `/`

### 5. 路由守卫（页面拦截）
- **位置**：`src/router/index.js`
- **拦截规则（最简版）**
  - 访问 `/login`：不拦截
  - 访问任何非 `/login` 页面：
    - 如果 `localStorage` 没有 `erp_token` → **强制跳转到** `/login?redirect=原地址`
    - 如果有 `erp_token` → 放行

### 6. 退出登录（Logout）
- **入口位置**：右上角头部（`src/layout/ErpLayout.vue`）
- **交互流程（干净利落的 3 步）**
  1. **清理现场**：清掉浏览器缓存里的 `token` 和用户信息
  2. **提示用户**：弹出“已退出登录”的成功提示
  3. **强制跳转**：立刻跳到 `/login`，不让用户停留在后台
- **为什么要二次确认？**
  - 防止误点：退出后会被踢回登录页，正在操作的数据可能丢失
- **清理缓存用到了什么？（超小白解释 clear()）**
  - `localStorage` / `sessionStorage` 是浏览器提供的“本地小仓库”
  - 里面存的是一对一对的 `key -> value`
  - **两种清理方式：**
    - `removeItem('key')`：只删某一个 key（推荐，不误删别的缓存）
    - `clear()`：把仓库里的所有 key 全部清空（最干净，但可能误删别的模块缓存）
  - 本项目默认使用 `removeItem` 精准删除 `erp_token/erp_user`，更安全

### 7. 修改密码（Change Password）
- **入口位置**：右上角用户名下拉菜单 → `修改密码`（`src/layout/ErpLayout.vue`）
- **前端交互**
  - 弹出 Dialog，包含：
    - 旧密码
    - 新密码
    - 确认新密码
  - 三个输入框都开启 `show-password`（可显示/隐藏）
  - 校验规则：
    - 新密码与确认新密码必须一致
- **后端接口**：PUT `/api/users/change-password`
  - **请求头（重要）**
    - `Authorization: Bearer <token>`
  - **请求体**
    - `oldPassword`：旧密码
    - `newPassword`：新密码
  - **后端逻辑（白话版）**
    1. 后端先用 `token` 找到“当前登录用户”的 `UserCode`
    2. 用 SQL 查出该用户的旧密码，验证旧密码是否正确
    3. 验证通过后，再更新密码
  - **关键安全点（超小白解释 WHERE）**
    - SQL 的这一句：`WHERE UserCode = @UserCode`
    - 意思是：**只更新工号等于当前登录工号的那一行**
    - 因为 `@UserCode` 来自“后端根据 token 解析出来的当前用户”，不是前端随便传的，所以能确保“只改到自己”
- **安全要求（必须）**
  - 修改密码成功后，前端会自动执行退出登录逻辑，强制回到 `/login`

### 8. 小白操作清单（改完怎么验证）
- **第 1 步：确认数据库有账号**
  - 用 Navicat/SSMS 看一下 `Sys_Users` 是否有一条 `Status=1` 的记录
  - 记住它的 `UserName` 和 `Password`
- **第 2 步：启动后端**
  - 如果你改了 `server/index.js`，需要重启后端服务
  - 你可以先访问：`GET /api/health` 看后端是否存活
- **第 3 步：启动前端并访问**
  - 直接访问一个后台地址（例如 `/system/operator`）
  - 预期：会跳到 `/login`
- **第 4 步：登录测试**
  - 输入正确工号/密码 → 预期：提示“登录成功”，进入后台
  - 输入不存在工号 → 预期：提示“工号不存在”
  - 输入禁用账号（Status=0）→ 预期：提示“账号已被禁用...”
  - 输入错误密码 → 预期：提示“密码错误”
 - **第 5 步：修改密码测试**
  - 登录后点击右上角用户名 → `修改密码`
  - 旧密码错误 → 预期：提示 `旧密码验证失败`
  - 修改成功 → 预期：提示 `密码修改成功，请重新登录`，并自动回到 `/login`

### 9. 已知问题 / 下一步计划（先写清楚，避免以后忘）
- **密码已升级为 bcrypt 加密存储**：v1.0.6 起新增/改密会强制写入 bcrypt 字符串（以 `$2b$` 开头）。
- **兼容旧明文测试数据**：为了不影响你现在数据库里 50+ 条明文测试记录，登录/改密校验会先判断：
  - 如果数据库里的 `Password` **以 `$2b$` 开头** → 认为是 bcrypt → 用 `bcrypt.compare()` 校验
  - 如果数据库里的 `Password` **不是以 `$2b$` 开头** → 认为是旧明文 → 允许直接字符串对比（仅兼容）
- **一旦用户修改密码或新增用户**：后端会用 `bcrypt.hash()` 先加密再入库，从此进入“加密模式”，不再保存明文。
- **后端 token 是内存版**：重启后端会失效；后续如果要做真正鉴权，需要把 token/会话做成可持久化方案，并让其他接口校验 token。

### 10. 安全说明（bcrypt + Salt 保姆级解释）
- **我们用的算法**：`bcrypt`
- **为什么不能把密码明文存数据库？**
  - 只要数据库被误导出/被误读，明文密码会直接泄露（风险极高）。
- **Salt（盐值）到底起什么作用？（用大白话说清楚）**
  - 你可以把“盐”理解为：给每个密码额外加一段**随机的搅拌料**。
  - 如果没有盐：同样的密码（比如很多人都用 `123456`），存到数据库里的结果会一模一样。
    - 这样别人一眼就能看出“哪些人密码一样”，甚至可以用现成的“常见密码字典”批量撞库。
  - 有了盐：即使两个人明文密码一样，bcrypt 也会因为 salt 不同，生成**完全不同**的加密字符串。
    - 这会让“批量破解”和“字典对照”变得非常困难。
  - **重要补充**：bcrypt 的结果字符串里已经包含了 salt 和成本参数，所以我们不需要单独建字段存 salt。
